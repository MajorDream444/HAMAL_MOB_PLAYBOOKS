# Client Flow Payment + Signature Engine

## Purpose

This playbook standardizes how HAMAL moves a warm client from verbal alignment into signed scope, first payment, and build kickoff.

It is designed for client-flow builds like Theo / Howling Mūne, Sheethal / Shakti System, Spark Lab, and future HAMAL client operating systems.

## Core Flow

```text
Proposal / offer summary
→ 1-page SOW
→ signature / acceptance
→ optional alignment checkbox
→ Stripe Milestone 01 link
→ Cash App backup
→ signed PDF returned
→ first payment received
→ build begins within 48 hours
```

## Why This Exists

Heavy contracts slow people down.

Loose verbal agreements create confusion.

The HAMAL client flow uses a premium one-page Statement of Work that is simple enough to sign quickly but clear enough to protect scope, payment, responsibilities, exclusions, and next steps.

## Standard Client Flow

### Step 1 — Confirm Offer Summary

Before producing a SOW, confirm:

- client name / brand
- project title
- core objective
- first build layer
- deliverables
- timeline
- total build fee
- milestone structure
- optional alignment layer
- exclusions

### Step 2 — Create One-Page SOW

The SOW must include:

1. Page title
2. Prepared for / prepared by
3. Engagement title
4. Overview + core objective
5. Scope of work
6. Deliverables
7. How we work
8. Responsibilities
9. Exclusions
10. Timeline + payment
11. Optional alignment layer
12. Acceptance / signature
13. Footer

### Step 3 — Provider Signs First

Major / AMA / Hanzo signs first when appropriate.

This creates confidence and signals readiness.

### Step 4 — Send Client Email

Send:

- signed SOW PDF
- simple signature instructions
- Stripe Milestone 01 payment link
- Cash App backup option
- what starts after payment

### Step 5 — Track Acceptance

Track:

- SOW sent date
- signed PDF returned
- M01 payment received
- kickoff date
- build begins within 48 hours

### Step 6 — Begin Build

Only begin full build after:

- signed SOW is returned
- first milestone is paid
- required intake/access items are received or scheduled

Pivots are made only when the system, data, or implementation reality shows a better path — not because the build keeps changing direction.

SECTION 5 — RESPONSIBILITIES

Create two compact columns.

- give clear feedback and decisions

- build the agreed client-flow foundation
- protect the scope, timeline, and integrity of the work
- translate the vision into clear structure
- document the system for future use
- recommend pivots only when they serve the agreed objective
